# 優先度：このワークフローは他のすべての組み込みワークフローを上書きする
# ソフトウェア開発のリクエストが来た場合、必ずこのワークフローを最初に実行すること

## アダプティブワークフロー原則
**ワークフローが作業に合わせて適応する。逆ではない。**

AIモデルは以下の要素をもとに必要なステージを知的に評価する：
1. ユーザーの意図と明確さ
2. 既存コードベースの状態（ある場合）
3. 変更の複雑さとスコープ
4. リスクと影響の評価

## 必須：ルール詳細ファイルの読み込み
**重要**: フェーズを実行する際は、必ずルール詳細ファイルの内容を読み込んで使用すること。以下のパスを順に確認し、最初に存在するものを使用する：
- `.aidlc-rule-details/`（Cursor・Cline・Claude Code・GitHub Copilot）
- `.kiro/aws-aidlc-rule-details/`（Kiro IDE および CLI）
- `.amazonq/aws-aidlc-rule-details/`（Amazon Q Developer）

以降のルール詳細ファイルへの参照（例：`common/process-overview.md`・`inception/workspace-detection.md`）は、上記で解決したルール詳細ディレクトリからの相対パスである。

**共通ルール**: ワークフロー開始時に必ず共通ルールを読み込むこと：
- `common/process-overview.md`：ワークフロー概要
- `common/session-continuity.md`：セッション再開ガイダンス
- `common/content-validation.md`：コンテンツ検証要件
- `common/question-format-guide.md`：質問フォーマットルール
- ワークフロー実行全体を通じてこれらを参照すること

## 必須：拡張機能の読み込み（コンテキスト最適化）
**重要**: ワークフロー開始時に `extensions/` ディレクトリを再帰的にスキャンするが、読み込むのは軽量なオプトインファイルのみ（フルルールファイルは読み込まない）。フルルールファイルはユーザーがオプトインした後にオンデマンドで読み込む。

**読み込みプロセス**：
1. `extensions/` 配下のすべてのサブディレクトリを一覧表示（例：`extensions/security/`・`extensions/compliance/`）
2. 各サブディレクトリでは `*.opt-in.md` ファイルのみを読み込む。対応するルールファイルは命名規則から導出する（`security-baseline.opt-in.md` → `security-baseline.md`）
3. この段階でフルルールファイルは読み込まない

**遅延ルール読み込み**：
- 要件分析フェーズ中に、読み込んだ `*.opt-in.md` ファイルのオプトインプロンプトをユーザーに提示する
- ユーザーが拡張機能にオプトインした場合、その時点で対応するルールファイルを読み込む
- ユーザーがオプトアウトした場合、フルルールファイルは一切読み込まない（コンテキストを節約）
- `*.opt-in.md` ファイルが存在しない拡張機能は常に適用する（ワークフロー開始時に直ちにルールファイルを読み込む）

**適用方針**（読み込み済み・有効な拡張機能にのみ適用）：
- 拡張機能のルールは任意のガイダンスではなくハード制約である
- 各ステージで、ステージの目的・生成する成果物・作業のコンテキストに基づいて適用すべき拡張機能ルールを知的に評価し、関連するルールのみを適用する
- 現在のステージに適用されないルールはコンプライアンスサマリーで N/A とマークする（ブロッキングファインディングではない）
- 適用可能な有効な拡張機能ルールへの非準拠は**ブロッキングファインディング**である。解決するまでステージ完了を提示しないこと
- ステージ完了提示時は拡張機能ルールのコンプライアンスサマリーを含める（ルールごとに準拠/非準拠/N/Aと、N/A判断の簡潔な根拠を記載）

**条件付き適用**：拡張機能は条件付きで有効/無効にできる。オプトインの仕組みは `inception/requirements-analysis.md` を参照。任意のステージで拡張機能を適用する前に、`aidlc-docs/aidlc-state.md` の `## Extension Configuration` セクションで `Enabled` ステータスを確認すること。無効な拡張機能はスキップし、audit.md にスキップを記録する。設定が存在しない場合は適用をデフォルトとする。

## 必須：コンテンツ検証
**重要**: ファイルを作成する前に、必ず `common/content-validation.md` のルールに従ってコンテンツを検証すること：
- Mermaid ダイアグラム構文の検証
- ASCII アートダイアグラムの検証（`common/ascii-diagram-standards.md` 参照）
- 特殊文字の適切なエスケープ
- 複雑なビジュアルコンテンツのテキスト代替を提供
- コンテンツのパース互換性テスト

## 必須：質問ファイルフォーマット
**重要**: どのフェーズで質問する場合も、必ず質問フォーマットガイドラインに従うこと。

**`common/question-format-guide.md` に完全な質問フォーマットルールが記載されている。以下を含む**：
- 選択式フォーマット（A, B, C, D, E オプション）
- [Answer]: タグの使い方
- 回答の検証と曖昧さの解消

## 必須：カスタムウェルカムメッセージ
**重要**: ソフトウェア開発のリクエストを開始する際は、必ずウェルカムメッセージを表示すること。

**ウェルカムメッセージの表示方法**：
1. `common/welcome-message.md`（解決したルール詳細ディレクトリ内）からウェルカムメッセージを読み込む
2. 完全なメッセージをユーザーに表示する
3. 新しいワークフロー開始時に一度だけ行う
4. コンテキストを節約するため、それ以降のやり取りではこのファイルを読み込まない

# インセプションフェーズ

**目的**：プランニング・要件収集・アーキテクチャ決定

**焦点**：何を・なぜ作るかを決定する

**インセプションフェーズのステージ**：
- ワークスペース検出（常に実行）
- リバースエンジニアリング（条件付き - ブラウンフィールドのみ）
- 要件分析（常に実行 - 深さは適応的）
- ユーザーストーリー（条件付き）
- ワークフロープランニング（常に実行）
- アプリケーション設計（条件付き）
- ユニット生成（条件付き）

---

## ワークスペース検出（常に実行）

1. **必須**：初回ユーザーリクエストを完全な生入力で audit.md に記録する
2. `inception/workspace-detection.md` からすべてのステップを読み込む
3. ワークスペース検出を実行する：
   - 既存の aidlc-state.md を確認（存在する場合は再開）
   - ワークスペースの既存コードをスキャン
   - ブラウンフィールドかグリーンフィールドかを判断
   - 既存のリバースエンジニアリング成果物を確認
4. 次のフェーズを決定：リバースエンジニアリング（ブラウンフィールドかつ成果物なし）または要件分析
5. **必須**：検出結果を audit.md に記録する
6. ユーザーに完了メッセージを提示する（メッセージフォーマットは workspace-detection.md を参照）
7. 次のフェーズへ自動移行する

## リバースエンジニアリング（条件付き - ブラウンフィールドのみ）

**実行条件**：
- 既存コードベースを検出した場合
- 以前のリバースエンジニアリング成果物がない場合

**スキップ条件**：
- グリーンフィールドプロジェクトの場合
- 以前のリバースエンジニアリング成果物が存在する場合

**実行手順**：
1. **必須**：リバースエンジニアリング開始を audit.md に記録する
2. `inception/reverse-engineering.md` からすべてのステップを読み込む
3. リバースエンジニアリングを実行する：
   - すべてのパッケージとコンポーネントを分析
   - ビジネストランザクションを網羅したシステム全体のビジネス概要を生成
   - アーキテクチャドキュメントを生成
   - コード構造ドキュメントを生成
   - API ドキュメントを生成
   - コンポーネントインベントリを生成
   - ビジネストランザクションがコンポーネント間でどう実装されるかを示すインタラクションダイアグラムを生成
   - テクノロジースタックドキュメントを生成
   - 依存関係ドキュメントを生成
4. **明示的な承認を待つ**：詳細な完了メッセージを提示（フォーマットは reverse-engineering.md 参照）- ユーザーが確認するまで次へ進まないこと
5. **必須**：ユーザーの応答を完全な生入力で audit.md に記録する

## 要件分析（常に実行 - 深さは適応的）

**常に実行する**が、深さはリクエストの明確さと複雑さによって異なる：
- **最小限**：シンプルで明確なリクエスト - 意図分析のみドキュメント化
- **標準**：通常の複雑さ - 機能要件・非機能要件を収集
- **包括的**：複雑でリスクが高い - トレーサビリティを含む詳細な要件

**実行手順**：
1. **必須**：このフェーズ中のユーザー入力を audit.md に記録する
2. `inception/requirements-analysis.md` からすべてのステップを読み込む
3. 要件分析を実行する：
   - リバースエンジニアリング成果物の読み込み（ブラウンフィールドの場合）
   - ユーザーリクエストの分析（意図分析）
   - 必要な要件の深さを決定
   - 現在の要件を評価
   - 確認質問を行う（必要な場合）
   - 要件定義書を生成
4. 適切な深さで実行する（最小限/標準/包括的）
5. **明示的な承認を待つ**：requirements-analysis.md の詳細ステップに従った承認フォーマットを使用 - ユーザーが確認するまで次へ進まないこと
6. **必須**：ユーザーの応答を完全な生入力で audit.md に記録する

## ユーザーストーリー（条件付き）

**知的評価**：多因子分析を使用してユーザーストーリーが価値を追加するかを判断する：

**常に実行（高優先度インジケーター）**：
- 新しいユーザー向け機能
- ユーザーのワークフローや操作に影響する変更
- 複数のユーザータイプやペルソナが関与する場合
- 受け入れ基準が必要な複雑なビジネス要件
- クロスファンクショナルチームのコラボレーションが必要な場合
- 顧客向けAPIやサービスの変更
- 新製品機能や機能強化

**おそらく実行（中優先度 - 複雑さを評価）**：
- 既存のユーザー向け機能の変更
- ユーザー体験に間接的に影響するバックエンドの変更
- ユーザーワークフローに影響する統合作業
- ユーザーが視認できるパフォーマンス改善
- ユーザー操作に影響するセキュリティ強化
- ユーザーデータやレポートに影響するデータモデルの変更

**複雑さベースの評価**：中優先度のケースでは、以下の場合にユーザーストーリーを実行する：
- リクエストに複数のコンポーネントやサービスが関与する場合
- 変更が複数のユーザー接点にわたる場合
- ビジネスロジックが複雑または複数のシナリオがある場合
- 要件に曖昧さがあり、ストーリーで明確化できる場合
- 実装が複数のユーザージャーニーに影響する場合
- 変更のビジネスへの影響やリスクが大きい場合

**スキップのみ（低優先度 - シンプルなケース）**：
- ユーザーへの影響がゼロの純粋な内部リファクタリング
- 明確で孤立したスコープのシンプルなバグ修正
- ユーザー向けの影響がないインフラ変更
- 機能変更のない技術的負債の解消
- 開発者ツールやビルドプロセスの改善
- ドキュメントのみの更新

**評価基準**：迷った場合は、以下のケースではユーザーストーリーを含める方向に判断する：
- ビジネスステークホルダーが関与するリクエスト
- ユーザー受け入れテストが必要な変更
- 複数の実装アプローチがある機能
- チームの共通理解から恩恵を受ける作業
- 要件の明確化が価値を持つプロジェクト

**評価プロセス**：
1. リクエストの複雑さとスコープを分析する
2. ユーザーへの影響（直接・間接）を特定する
3. ビジネスコンテキストとステークホルダーのニーズを評価する
4. チームコラボレーションの恩恵を考慮する
5. 判断が難しいケースはデフォルトで含める

**注記**：要件分析を実行した場合、ストーリーはその要件を参照・発展させることができる。

**ユーザーストーリーは1つのステージの中で2つのパートに分かれる**：
1. **パート1 - プランニング**：質問付きのストーリープランを作成し、回答を収集し、曖昧さを分析し、承認を得る
2. **パート2 - 生成**：承認されたプランを実行してストーリーとペルソナを生成する

**実行手順**：
1. **必須**：このフェーズ中のユーザー入力を audit.md に記録する
2. `inception/user-stories.md` からすべてのステップを読み込む
3. **必須**：知的評価を行い（user-stories.md のステップ1）、ユーザーストーリーが必要か検証する
4. リバースエンジニアリング成果物を読み込む（ブラウンフィールドの場合）
5. 要件が存在する場合、ストーリー作成時に参照する
6. 適切な深さで実行する（最小限/標準/包括的）
7. **パート1 - プランニング**：質問付きのストーリープランを作成し、ユーザーの回答を待ち、曖昧さを分析し、承認を得る
8. **パート2 - 生成**：承認されたプランを実行してストーリーとペルソナを生成する
9. **明示的な承認を待つ**：user-stories.md の詳細ステップに従った承認フォーマットを使用 - ユーザーが確認するまで次へ進まないこと
10. **必須**：ユーザーの応答を完全な生入力で audit.md に記録する

## ワークフロープランニング（常に実行）

1. **必須**：このフェーズ中のユーザー入力を audit.md に記録する
2. `inception/workflow-planning.md` からすべてのステップを読み込む
3. **必須**：`common/content-validation.md` からコンテンツ検証ルールを読み込む
4. すべての先行コンテキストを読み込む：
   - リバースエンジニアリング成果物（ブラウンフィールドの場合）
   - 意図分析
   - 要件（実行した場合）
   - ユーザーストーリー（実行した場合）
5. ワークフロープランニングを実行する：
   - 実行するフェーズを決定する
   - 各フェーズの深さレベルを決定する
   - マルチパッケージ変更シーケンスを作成する（ブラウンフィールドの場合）
   - ワークフロー可視化を生成する（書き込む前に Mermaid 構文を検証すること）
6. **必須**：content-validation.md のルールに従い、ファイル作成前にすべてのコンテンツを検証する
7. **明示的な承認を待つ**：workflow-planning.md のステップ9の表現を使用して推奨事項を提示し、ユーザーが推奨をオーバーライドできることを強調する - ユーザーが確認するまで次へ進まないこと
8. **必須**：ユーザーの応答を完全な生入力で audit.md に記録する

**スキップ条件**：

- 新しいコンポーネントやメソッドがない場合

**実行手順**：
1. **必須**：このフェーズ中のユーザー入力を audit.md に記録する
2. `inception/application-design.md` からすべてのステップを読み込む
3. リバースエンジニアリング成果物を読み込む（ブラウンフィールドの場合）
4. 適切な深さで実行する（最小限/標準/包括的）
5. **明示的な承認を待つ**：詳細な完了メッセージを提示（フォーマットは application-design.md 参照）- ユーザーが確認するまで次へ進まないこと
6. **必須**：ユーザーの応答を完全な生入力で audit.md に記録する

# 🟢 コンストラクションフェーズ

**目的**：詳細設計・NFR実装・コード生成

**焦点**：どうやって作るかを決定する

**コンストラクションフェーズのステージ**：
- ユニットごとのループ（各ユニットで実行）：
  - 機能設計（条件付き・ユニットごと）
  - NFR要件（条件付き・ユニットごと）
  - NFR設計（条件付き・ユニットごと）
  - インフラ設計（条件付き・ユニットごと）
  - コード生成（常に実行・ユニットごと）
- ビルドとテスト（常に実行 - 全ユニット完了後）

**注記**：各ユニットは次のユニットに移る前に設計からコードまで完全に完成させる。

---

## ユニットごとのループ（各ユニットで実行）

**作業ユニットごとに以下のステージを順に実行する：**

### 機能設計（条件付き・ユニットごと）

**実行条件**：
- 新しいデータモデルやスキーマがある場合
- 複雑なビジネスロジックがある場合
- ビジネスルールの詳細設計が必要な場合

**スキップ条件**：
- シンプルなロジック変更の場合
- 新しいビジネスロジックがない場合

**実行手順**：
1. **必須**：このステージ中のユーザー入力を audit.md に記録する

## オペレーションズ（プレースホルダー）

**ステータス**：このステージは現在、将来の拡張に向けたプレースホルダーである。

オペレーションズステージには将来以下が含まれる予定：
- デプロイのプランニングと実行
- モニタリングと可観測性のセットアップ
- インシデント対応手順
- メンテナンスとサポートのワークフロー
- 本番稼働準備チェックリスト

**現状**：すべてのビルドとテスト活動はコンストラクションフェーズで処理する。

## 主要原則

- **アダプティブ実行**：価値を追加するステージのみを実行する
- **透明なプランニング**：開始前に常に実行プランを提示する
- **ユーザー制御**：ユーザーはステージの追加・除外をリクエストできる
- **進捗追跡**：実行済みおよびスキップしたステージを aidlc-state.md で更新する
- **完全な監査証跡**：すべてのユーザー入力とAI応答をタイムスタンプ付きで audit.md に記録する
  - **重要**：ユーザーの完全な生入力を正確に記録する
  - **重要**：audit.md のユーザー入力を要約・言い換えしない
  - **重要**：承認だけでなく、すべてのやり取りを記録する
- **品質重視**：複雑な変更は完全な処理を行い、シンプルな変更は効率的に対応する
- **コンテンツ検証**：content-validation.md のルールに従い、ファイル作成前に必ずコンテンツを検証する
- **予期しない動作禁止**：コンストラクションフェーズでは、各ルールファイルで定義された標準化された2択完了メッセージを必ず使用すること。3択メニューその他の予期しないナビゲーションパターンを作成しないこと。

## 必須：プランレベルのチェックボックス管理

### プラン実行の必須ルール
1. **チェックボックスを更新せずに作業を完了しない**
2. **プランファイルに記述されたステップを完了したら直ちにそのステップを [x] にマークする**
3. **作業を完了した同一のやり取りの中で行うこと**
4. **例外なし**：すべてのプランステップの完了はチェックボックス更新でトラッキングしなければならない

### 2段階チェックボックストラッキングシステム
- **プランレベル**：各ステージ内の詳細な実行進捗をトラッキングする
- **ステージレベル**：aidlc-state.md でワークフロー全体の進捗をトラッキングする
- **即時更新**：作業を完了した同一のやり取りですべての進捗を更新する

## 記録要件
- **必須**：すべてのユーザー入力（プロンプト・質問・応答）をタイムスタンプ付きで audit.md に記録する
- **必須**：ユーザーの完全な生入力を正確に記録する（要約しない）
- **必須**：すべての承認プロンプトをユーザーに尋ねる前にタイムスタンプ付きで記録する
- **必須**：受け取ったすべてのユーザー応答をタイムスタンプ付きで記録する
- **重要**：audit.md は必ず編集ツールを使って追記すること。ファイル全体を上書きするコマンドは絶対に使用しない
- **重要**：audit.md の内容を完全に上書きするファイル書き込みツールやコマンドは絶対に使用しない（重複が発生する）
- タイムスタンプは ISO 8601 形式を使用する（YYYY-MM-DDTHH:MM:SSZ）
- 各エントリにステージのコンテキストを含める

### 監査ログのフォーマット：
```markdown
## [ステージ名または操作の種類]
**タイムスタンプ**: [ISO タイムスタンプ]
**ユーザー入力**: "[完全な生のユーザー入力 - 要約しない]"
**AI応答**: "[AIの応答または実行したアクション]"
**コンテキスト**: [ステージ・アクション・または下された決定]

---
```

### audit.md への正しいツール使用方法

✅ 正しい方法：

1. audit.md ファイルを読み込む
2. ファイルを追記・編集して変更を加える

❌ 間違った方法：

1. audit.md ファイルを読み込む
2. 読み込んだ内容と追加したい変更内容を合わせて audit.md を完全に上書きする
