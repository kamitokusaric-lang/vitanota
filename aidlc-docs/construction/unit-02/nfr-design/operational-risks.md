## 目的

## スコア定義

- **深刻度**: 🔴高（サービス停止・データ漏えい等） / 🟡中（性能低下・運用負荷増） / 🟢低（通知・監視の不足程度）
- **発生確率**: 高（数週間以内に発生想定） / 中（数ヶ月以内） / 低（年1回未満）

## R1: RDS Proxy セッションピンニング（🔴 高）

**問題**:
RLS 用に `SET LOCAL app.tenant_id` / `app.user_id` をトランザクション内で設定するが、RDS Proxy のドライバ検知ロジックによっては接続がピン（pin）され、多重化が機能しなくなる可能性がある。

**影響**:
- RDS 接続数が急増し `max_connections` に到達 → 503 多発
- App Runner のスケールアウトが無駄になる

**緩和策**:
- [ ] `SET LOCAL` は必ず `db.transaction` 内で実行（Drizzle のトランザクション境界内に限定）
- [ ] セッションレベルの `SET`（LOCAL なし）は禁止
- [ ] コードレビュー時のチェックリスト項目化
- [ ] Unit-02 統合テストで検証

**検証方法**:
```sql
-- 統合テスト環境で実行
SELECT name, count, (DatabaseConnectionsCurrentlySessionPinned / count * 100) AS pin_ratio
FROM pg_stat_database;
```
- CloudWatch の `DatabaseConnectionsCurrentlySessionPinned` メトリクスで常時監視
- ピン率 > 5% でアラート発報

**フォールバック**:
ピンが多発する場合、RDS Proxy を外して Drizzle/pg 側の接続プールに切替。その場合は接続数の上限管理をアプリ側で厳密に行う。

## R4: CloudFront + SWR の二重キャッシュ

**問題**:
Unit-01 で CloudFront を前段に配置したため、`Cache-Control: s-maxage=30, stale-while-revalidate=60`（NFR-U02-02）はエッジでも効く。加えてクライアント側 SWR ライブラリが独自にキャッシュする。エントリ投稿後の `mutate()` ではクライアント層の SWR キャッシュしか無効化されず、**CloudFront エッジキャッシュは最大 90秒間古いまま**。

**影響シナリオ**:
1. 教員A が 10:00:00 にエントリ投稿
2. 教員B が 10:00:05 にタイムラインを開く
3. CloudFront エッジキャッシュから古い（10:00:00 以前の）レスポンスを返却
4. 最大 10:01:30 まで新しいエントリが見えない

**緩和策**:
- [ ] 共有タイムライン用のキャッシュキーに `user_id` を含めない（テナント内共通キャッシュとする）
- [ ] **エントリ作成/更新/削除後は CloudFront invalidation を API 側から発行**
  - パフォーマンス/コスト懸念あり（1000件/月まで無料、それ以上は $0.005/path）
  - 代替: `Surrogate-Key` ヘッダー + Lambda@Edge での選択的無効化（オーバースペック）
- [ ] **推奨**: `s-maxage=30`（30秒の遅延は教員日誌 UX では許容範囲）と割り切り、SWR エッジキャッシュ無効化は実装しない
- [ ] クライアント側は SWR `mutate()` で即時更新して楽観的表示、エッジキャッシュの更新は最大90秒の遅延を受容

**設計判断**: NFR-U02-02 の本来の意図（RDS 負荷軽減）を優先し、30秒遅延を許容する方針で確定する。

**追加強化（SP-U02-04 多層防御）**:
- `/api/public/*` と `/api/private/*` のパス名前空間分離により、CloudFront キャッシュ対象の混入事故を構造的に排除
- `public_journal_entries` VIEW が `is_public=true` のみ露出することで、仮にアプリバグで誤ったレスポンスを返しても View が物理的に非公開エントリをブロック
- **論点 F（PII エッジキャッシュ混入リスク）は多層防御により実質的に解消**

## R5: RLS ポリシー順序バグ

**問題**:
`journal_entries` に 2ポリシー（public_read + owner_all）を PERMISSIVE で配置すると OR 結合される。設計意図通りの挙動になるが、RESTRICTIVE と誤って設定するとアクセス不能になる。また、セッション変数未設定時の挙動も要確認。

**緩和策**:
- [ ] 両ポリシーとも `AS PERMISSIVE` を明示
- [ ] セッション変数未設定時（`current_setting` の missing_ok=true 時）は NULL → 比較が失敗して全拒否となる fail-safe を確認

**テスト要件（NFR-U02-07 拡張）**:
- [ ] テナントA・教員Xの公開エントリをテナントA・教員Yが読める
- [ ] テナントA・教員Xの非公開エントリをテナントA・教員Yが読めない
- [ ] テナントA・教員Xの公開エントリをテナントB・教員Zが読めない
- [ ] テナントA・教員Xの非公開エントリをテナントB・教員Zが読めない
- [ ] セッション変数未設定時に全ての SELECT が 0件
- [ ] 実 PostgreSQL で検証（モック不可）

## R7: ローリングデプロイ中のマイグレーション競合

**問題**:
App Runner のローリングデプロイ中、旧バージョンと新バージョンが**同時に DB に接続する期間**がある。新バージョンのマイグレーションが旧バージョンのクエリを破壊するスキーマ変更を含むと、旧インスタンスで 500 エラー多発。

**緩和策（標準）**:
- [ ] マイグレーションは**後方互換性を維持**（NOT NULL カラム追加は段階的、列削除は2段階デプロイ）
- [ ] マイグレーション実行タイミング: デプロイパイプラインの App Runner 更新**前**に GitHub Actions で実行
- [ ] ロールバック時に備え、直前のマイグレーションを逆方向適用可能にする
- [ ] 破壊的変更を含むマイグレーションはメンテナンスウィンドウで実施

## R11: CloudFront → App Runner 署名ヘッダーローテーション事故

**問題**:
`X-CloudFront-Secret` ヘッダーをローテーションする際、CloudFront 側と App Runner 側の更新タイミングがずれると全リクエストが 403 になる。

**緩和策**:
- [ ] Secrets Manager で**新旧2値を一時的に保持**（`current` / `previous`）
- [ ] App Runner は両方を許容するミドルウェア実装
- [ ] CloudFront を新値に更新後、24時間経過してから Secrets Manager の `previous` を削除
- [ ] ローテーション手順を Runbook 化

---

## R12: WAF 誤検知による正当リクエストブロック

**問題**:
AWS Managed Rules は一般的な攻撃パターンをブロックするが、日誌本文に「誤爆ワード」が含まれると 403 となる可能性（例: `<script>` という文字列を含む教員メモ）。

**緩和策**:
- [ ] WAF ログを CloudWatch Logs に出力
- [ ] `SampledRequests` を有効化し、BlockedRequests のサンプルを定期レビュー
- [ ] 誤検知が発見されたルールは**除外条件を追加**（パス指定・特定フィールドでルール無効化）
- [ ] Unit-02 の日誌エントリ投稿エンドポイント `/api/journal/entries` は、初期はルールを **Count モード**で運用し、実データでチューニング後に Block に切替
