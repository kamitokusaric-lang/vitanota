## 1. 目的

教員が日々ノートの投稿をもとに、**直前完了週 (= 先週) を無理なく振り返れるようにすること**。

- 評価しない
- 分析しすぎない
- 安心して書ける状態を保つ
- 寄り添いすぎず・冷たすぎず、友達のように

## 2. 設計思想 (踏み絵)

> AI は「見抜く存在」ではなく、「そっと言い換える存在」。
> AI は「分析」ではなく「ねぎらい」を行う。
> AI は **観察者ではなく、友達視点でドライに労うアシスタント**。

- 個人を評価しない
- 観測されてる感をゼロに保つ (= memory `feedback_observed_moment_broken.md` の踏み絵を死守)
- 投稿の安心感を最優先
- **寄り添いすぎない / 心理推察しない / 諭しを引かない / 共感確認しない / 一般化で突き放さない**
  - これらが満たされないと「気持ち悪さ」「観察されてる感」が出る (実装中に複数回判明)

## 3. 機能概要

- 日々ノートの投稿 + タスク状況をもとに、**集計数字 (DB から直接) + AI ねぎらいコメント**を表示
- 表示対象は **本人のみ** (school_admin にも他教員にも見えない)
- 表示名: **「先週のvitanotaレポート」**
- ダッシュボード大タブの 1 つ (現在: 日々ノート / タスクボード / **先週のvitanotaレポート** / 時間割 / (admin) 学校全体の温度)

## 4. 生成タイミング

- **その週の初回アクセス時に自動生成**
- 同じ週の 2 回目以降のアクセスは保存済みを返す (再生成しない)
- 1 ユーザー × 1 週 = 1 回の生成
- 「週」の定義: **月曜始まり** (月〜日の 7 日間、`week_start` = 月曜の日付)
- **集計対象は「先週」(直前完了週)** — 月曜朝に開いて「先週どうだった?」のレポートが届く形

## 5. データフロー

```
[投稿時]
① POST /api/private/journal/entries
   ↓
② 前処理 (正規表現マスキング)
   ↓
③ DB 保存 (content + content_masked 両方)

[週次レポート要求時]
④ GET /api/me/weekly-summary
   ↓
⑤ Promise.all 並行:
   (a) 既存 summary レコードあり? → なければ AI 生成 → INSERT (race 対策で onConflictDoNothing)
   (b) 集計 stats を取得 (常に最新)
   ↓
⑥ レスポンス: { summary, weekStart, generatedAt, stats }
   ↓
⑦ UI: 数字表 (stats) + AI コメント (summary) を 2 層で表示
```

## 6. マスキング (投稿時の前処理)

### 6.1 方針

- 投稿時に **正規表現マスキング** を実行 (LLM 不使用、コスト 0、レイテンシ 0)
- DB 保存時点で `content_masked` カラムに格納
- 本人表示は `content` (原文)、AI 入力は `content_masked` のみ使用
- **on-the-fly mask**: 既存投稿で `content_masked` が NULL の場合、AI 入力時に動的に maskContent を呼ぶ (= migration 既存データへの移行コスト回避)

### 6.2 マスキングルール

| 対象 | 置換先 |
|---|---|
| ○○くん / 君 / ちゃん | 生徒 |
| ○○先生 / 教諭 | 同僚 |
| 校長 / 教頭 / 主任 (接頭辞オプショナル) | 同僚 |
| ○○さん / 様 | ある人 |
| ○年○組 / 数字組 | クラス |
| ○○小学校 / 中学校 / 高校 / 学園 | 学校 |

### 6.3 正規表現の設計判断

接頭辞は **「漢字・カタカナのみ」(ひらがな除外)** + **{2,4} 文字必須** で:
- ひらがな除外 → 助詞 (の/と/が) で greedy match が止まる、「3年2組の太郎くん」のような文での過剰検出を回避
- {2,4} 必須 → 「皆さん」「南先生」のような漢字 1 文字一般語の巻き込みを防止
- 副作用: 1 文字苗字 (「南さん」「翼くん」) は漏れる、許容範囲

役職 (校長/教頭/主任) は単独でも人を指すため `{0,4}` で接頭辞オプショナル。

### 6.4 実装場所

- `src/features/journal/lib/mask-content.ts` (正規表現セット + maskContent 関数)
- `__tests__/unit/mask-content.test.ts` (18 ケース、全 pass)
- `src/features/journal/lib/privateJournalRepository.ts` (POST/PUT 時に呼出し)

## 7. 集計データ

週次レポート時に DB から集計し、**UI 表示用 stats** と **AI 入力用 data** の 2 形式で取得:

### 7.1 UI 表示用 (WeeklySummaryStats)

```ts
interface WeeklySummaryStats {
  weekStart: string;     // YYYY-MM-DD (月曜)
  weekEnd: string;       // YYYY-MM-DD (日曜)
  entryCount: number;
  publicCount: number;
  privateCount: number;
  moodDistribution: Record<MoodLevel, number>;
  topTags: Array<{ name: string; count: number }>;
  tasksCompleted: number;
  tasksActive: number;
  tasksDueThisWeek: number;
  tasksAssignedFromOthers: number;
  tasksDelegatedToOthers: number;
}
```

UI で表 (dl/dt/dd) として表示。投稿テキストは含めない。

### 7.2 AI 入力用 (内部のみ)

UI 表示用に加えて `entries: Array<{ contentMasked, mood, isPublic }>` を含む。マスク済み投稿テキストを AI に渡す。

### 7.3 集計の正確性

- **完了**: `status='done' AND updatedAt が先週内` で近似 (tasks に completed_at カラムがあるが、本実装では updatedAt で代用、Phase 2 で precise 化検討)
- **動きあり**: createdAt OR updatedAt が先週内
- **期限到来**: dueDate が先週内 (DATE 型を YYYY-MM-DD 文字列で比較し TZ 問題回避)
- **依頼を受けた**: 新規 / createdBy ≠ 自分 & assignee = 自分 / createdAt が先週内
- **他人へ依頼**: 新規 / createdBy = 自分 & assignee ≠ 自分 / createdAt が先週内

scope は `owner_user_id = self OR created_by = self` で限定。

## 8. AI プロンプト (1 段階統合版)

### 8.1 設計の核心

AI の役割を **「事実報告 + ねぎらい両方書く」 → 「コメント + ねぎらい だけ」** に縮小。事実 (数字) は UI 側で表示するため、AI は「友達視点のひと言コメント」と「ストレートな労い」だけに専念。

### 8.2 出力構造 (厳守)

```
[第 1 段落 (1〜2 文)]
投稿から読み取った傾向を友達視点でひとこと
(客観的・ドライ、寄り添いすぎず)

[第 2 段落 (1 行で OK)]
短くストレートな労い (バリエーション持つ)
```

全体 100〜180 文字、2 段落。

### 8.3 出力ルール (絶対)

1. **見出し・タイトル・前置きを一切書かない** (本文のみ)
2. 個人を評価しない
3. 原文を引用しない、言葉を真似ない
4. マスク後の語 (生徒/同僚/ある人/クラス/学校) も文中に出さない
5. 数字や統計を文章に持ち込まない
6. 断定を避ける (推測形 ~ようです / ~みたいですね を使う)
7. アドバイス・指示・命令をしない (✗「~してください」)
8. **諭し・人生訓・教訓を一切引かない** (✗「そういう歩み方もある」「~こそが大事」「~でも充分です」)
9. **対比構造で意味付けしない** (✗「~ではなく、~」)
10. **「~です」「~と思います」「~なのだと思います」みたいな宣言・主張を避ける**
11. **寄り添いすぎない / 心理状態を推測しない / 内面を推察しない / 現在地を断定しない**
    - ✗「~気持ち」「~感じていた」「~を抱えていた」 (内面推察)
    - ✗「~流れの中にいる」「~状態にある」 (現在地断定)
    - ✗「~んですね」「~ですよね」「~わかります」 (共感確認 = 寄り添いすぎ = **気持ち悪さの最大の源泉**)
    - ○「~様子が伝わります」「~があったようです」 (外面観察に留める)
    - **AI は解釈する人ではなく、友達視点で映す鏡**
12. **二重ねぎらい禁止** (締めの「お疲れさま」は文末 1 回のみ)
13. 問題指摘ゼロ、見守る姿勢
14. 一週間全体に声をかける
15. 励ましじゃなく、認める / 受け止める / そばにいる
16. 投稿が少ない週でも、書かなかったことを問題視しない

### 8.4 トーン

- **客観的な事実描写 + 友達視点のドライなねぎらい、それだけ**
- 友達感、ストレート、ちょっと突き放し気味でも OK (= 鏡、解釈する人にはならない)
- 推奨語尾: 「~ね」「~だね」「~みたい」「~ようだ」「~よね」
- 推奨締め (バリエーション持つ):
  - 「お疲れさま」「お疲れ様」「お疲れさまでした」
  - 「今週もやっていこう」
  - 「お疲れさま。今週はぼちぼちいこう」
  - 「無理ない範囲で。今週もいい一週間に」
  - 「ひと息ついて、また今週も」
- 推奨語彙: 「ひと息」「結構抱えてた」「断れなかった」「片付けてる」「~みたい」「~ようだね」

### 8.5 避ける語彙 (累積)

| カテゴリ | 語彙 |
|---|---|
| 評価・励まし | 「素晴らしい」「凄い」「頑張って」「~するべき」「分析」「成長」 |
| 諭し・教訓 | 「~でいい」「~充分」「~こそが」「~ではなく」 |
| 命令・指示 | 「~してください」「~いってください」「過ごしていって」 |
| 内面推察 | 「~気持ち」「~感じていた」「~を抱えていた」 |
| 現在地断定 | 「~流れの中にいる」「~なかにいる」「~状態にある」 |
| 共感確認 (寄り添いすぎ) | 「~んですね」「~ですよね」「~わかります」 |
| 過剰な丁寧さ・文学的 | 「先生に」「歩んできた」「重ねてきた」「されてきた」 |
| 一般化突き放し (冷たい) | 「まあ、~でしょ」「そういう週もある」「~もあるでしょ」「~んじゃない」 |
| 締めの短縮 | 「お疲れ。」「お疲れ」 (= 必ず「お疲れさま」を使う) |

### 8.6 サーバー側 fallback ガードレール

AI 出力後、以下を check:

| Check | 動作 |
|---|---|
| 出力冒頭にタイトル行 (「vitanotaレポート」「ひとこと」「振り返り」を含む 30 字未満の行) | **自動剥がし** + 直後の空行除去 |
| 出力に「あなた」「君」 | 「先生」に置換 |
| 700 文字超 | 自動切り詰め (改行で切る、最低 400 字保持) |
| 50 文字未満 | fallback テンプレート表示 |

### 8.7 fallback テンプレート (生成失敗時)

```
今週も、一週間お疲れさまでした。

書いたこと、手を動かしたこと、その全部がここにあります。

少し、ひと息ついてくださいね。
```

## 9. データ保存

### 9.1 新規テーブル

```sql
CREATE TABLE journal_weekly_summaries (
  tenant_id    uuid        NOT NULL REFERENCES tenants(id) ON DELETE CASCADE,
  user_id      uuid        NOT NULL REFERENCES users(id)   ON DELETE CASCADE,
  week_start   date        NOT NULL,  -- 月曜日の日付 (= 直前完了週)
  summary      text        NOT NULL,
  generated_at timestamptz NOT NULL DEFAULT now(),
  PRIMARY KEY (user_id, week_start)
);
```

RLS: 本人のみアクセス可 (system_admin のみ bypass)。INSERT/UPDATE/DELETE 全て本人限定。

### 9.2 既存テーブルへのカラム追加

```sql
ALTER TABLE journal_entries ADD COLUMN content_masked TEXT;
```

### 9.3 race condition 対策

並行リクエスト (リロード連打 / SWR retry 等) で同じ (user_id, week_start) に複数 INSERT が走ると PK 違反 (23505) で 500。対策:

```ts
const [inserted] = await tx
  .insert(journalWeeklySummaries)
  .values({...})
  .onConflictDoNothing()
  .returning();

if (inserted) return inserted;

// 並行で他方が先に INSERT 済み → 既存を SELECT
const [existing] = await tx.select().from(...).where(...);
return existing;
```

副作用: 並行発生時に AI 呼出しが両方走る (= コスト 2 倍、頻度低い前提で許容)。

### 9.4 既存投稿の content_masked

MVP では **on-the-fly mask** で対応 (AI 入力時に NULL なら maskContent を呼ぶ)。バッチ backfill は post-mvp-backlog.md に記載。

## 11. AI モデル選定

- **Claude Haiku 4.5** (`claude-haiku-4-5-20251001`)
- 理由: トーン制御しやすい、低コスト、低レイテンシ
- max_tokens: 1024 (実出力は ~200 token = 日本語 130 字程度)

## 12. コスト試算

| 規模 | 試算 (月) | 試算 (年) |
|---|---|---|
| 1 校 50 名 × 週 1 回 | ¥48 | ¥600 |
| 10 校 500 名 × 週 1 回 | ¥480 | ¥6,000 |
| 100 校 5000 名 × 週 1 回 | ¥4,800 | ¥58,000 |

文字数を 200-300 → 100-180 に絞ったため、試算より少し下振れ。

## 14. NG 事項 (絶対やらない)

- 個人評価 (例:「あなたは〜」)
- 問題指摘 (例:「課題があります」)
- 原文引用
- 名前の出力
- 強いアドバイス
- スコア化、ランキング、前週比較、「成長」「改善」判定、他人比較
- 諭し・人生訓・寄り添い確認・一般化突き放し・短縮形「お疲れ」
