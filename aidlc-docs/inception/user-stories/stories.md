# vitanota — ユーザーストーリー

## 整理方針
- **構造**: ペルソナ × エピック（大機能単位）× ストーリー
- **優先度**: MoSCoW法（Must / Should / Could / Won't）
- **受け入れ基準**: 主要ストーリーは Given/When/Then、シンプルなものは箇条書き
- **言語**: 日本語

## MoSCoW 凡例
| 記号 | 意味 |
|---|---|
| 🔴 Must | MVPに必須。これがなければリリースできない |
| 🟡 Should | 重要だが、MVP直後のリリースでも許容できる |
| 🟢 Could | あれば価値があるが、なくてもよい |
| ⚫ Won't | フェーズ2以降に実装する |

## EPIC-T-01: 認証・アカウント

### US-T-001: Googleアカウントでログインする 🔴 Must

**ストーリー**: 私は教員として、Googleアカウントでログインしたい。なぜなら、別途パスワードを管理する手間なく安全にサービスを利用したいから。

**受け入れ基準**:
- GoogleログインボタンをクリックするとGoogle認証画面に遷移すること
- 認証成功後、自分のテナント（学校）の教員ダッシュボードにリダイレクトされること
- 他校のデータにはアクセスできないこと
- 認証失敗時は適切なエラーメッセージが表示されること

### US-T-002: ログアウトする 🔴 Must

**ストーリー**: 私は教員として、セキュアにログアウトしたい。なぜなら、共有端末での情報漏洩を防ぎたいから。

**受け入れ基準**:
- ログアウト後、セッションが即座に無効化されること
- ログアウト後にブラウザの戻るボタンを押しても認証済みページにアクセスできないこと

### US-T-003: AI週次レポートへの同意を設定する 🟡 Should

**ストーリー**: 私は教員として、AIレポート機能への同意・拒否を設定したい。なぜなら、自分のデータがAI処理に使われることを理解したうえで判断したいから。

**受け入れ基準**:
- 初回ログイン時に同意確認画面が表示されること
- 同意しない場合でも、AI週次レポート以外の機能は全て利用できること
- 設定はいつでも変更できること

## EPIC-T-02: 日誌記録

### US-T-010: 日誌エントリを作成する 🔴 Must

**ストーリー**: 私は教員として、今日の出来事や気持ちをプレーンテキストで記録したい。なぜなら、日々の変化を後から振り返れるようにしたいから。

**受け入れ基準**（2026-04-15 改訂: 共有タイムライン方式・巻末の変更履歴を参照）:
- Given: 教員が `/journal` にログインしている
- When: 「新規投稿」ボタンをクリックしてテキストを入力し保存する
- Then: マイ記録（`/journal/mine`）の先頭に新しいエントリが表示され、作成日時が記録される
- Then: 「自分だけに保存」を**チェックしなかった**場合、共有タイムライン（`/journal`）の先頭にも表示され、テナント内の他教員からも閲覧可能となる
- Then: 「自分だけに保存」を**チェックした**場合、共有タイムラインには表示されずマイ記録のみに残る

- Given: 1日に複数回記録したい
- When: 同じ日に2件目のエントリを作成する
- Then: 同じ日付で複数のエントリが時系列で表示される

- Given: 本文は 1〜200 文字の範囲で入力する
- When: 空文字もしくは 201 文字以上で保存しようとする
- Then: クライアント側バリデーションでエラー表示され、送信されない

### US-T-011: 日誌エントリを編集する 🔴 Must

**ストーリー**: 私は教員として、記録した内容を後から修正したい。なぜなら、書き間違いや書き足しが発生するから。

**受け入れ基準**:
- 自分のエントリのみ編集できること（他の教員のエントリは編集不可、URL 直打ちでも 404 が返る）
- 編集後、`updated_at` が現在時刻に更新されること
- 編集ページ（`/journal/[id]/edit`）はマイ記録からの編集リンク経由でのみ到達する導線
- 公開設定（is_public）も編集時に変更可能

### US-T-012: 日誌エントリを削除する 🔴 Must

**ストーリー**: 私は教員として、記録したエントリを削除したい。なぜなら、誤って記録した内容を残したくないから。

**受け入れ基準**:
- 削除前にブラウザの `confirm()` ダイアログで確認を求めること
- 自分のエントリのみ削除できること（他の教員のエントリは URL 直打ちでも 404）
- 削除後、エントリに紐づいていた `journal_entry_tags` も自動削除されること（タグ本体は残る）

### US-T-013: エントリにタグを付ける 🔴 Must

**ストーリー**: 私は教員として、エントリにタグを付けて分類したい。なぜなら、後からどんな出来事が多かったかを把握したいから。

**受け入れ基準**（2026-04-15 改訂: タグ統合・巻末の変更履歴を参照）:
- 1つのエントリに最大10件までのタグを付与できること
- 既存タグから選択できること（テナント内の全タグがボタン表示される）
- タグには感情タグ（`is_emotion=true`）と業務タグ（`is_emotion=false`）があり、UI 上で視覚的に区別されること
- テナント作成時にシステムデフォルトタグ8件（感情5・業務3）が自動シードされていること
- タグが20件を超えた場合、テキスト入力による絞り込みができること
- 新しいタグを教員が作成できること（school_admin のみが削除可能）

### US-T-014: タイムラインで日誌を閲覧する 🔴 Must

**ストーリー**: 私は教員として、テナント内の教員全員で日誌をタイムライン形式で共有閲覧したい。なぜなら、同じ学校内での気づきや工夫を共有して、互いに学び合いたいから。

**受け入れ基準**（2026-04-15 改訂: 共有タイムライン方式に更新・巻末の変更履歴を参照）:

**共有タイムライン（`/journal`）**:
- Given: 教員が `/journal` を開いている
- When: タイムラインが表示される
- Then: 同一テナントの全教員の **公開エントリ**（`is_public=true`）が新しい順に20件/ページで表示される
- Then: 各エントリは投稿者名・日時・本文冒頭50文字・タグを表示する
- Then: **非公開エントリ（`is_public=false`）は絶対に表示されない**（SP-U02-04 8層防御）
- Then: 別テナントの教員のエントリは一切表示されない（RLS + 複合 FK）

**マイ記録（`/journal/mine`）**:
- Given: 教員が `/journal/mine` を開いている
- When: マイ記録が表示される
- Then: 自分の全エントリ（公開・非公開両方）が新しい順に表示される
- Then: 非公開エントリには「自分だけに保存」バッジが表示される
- Then: 各エントリから編集ページに遷移できる

## EPIC-T-03: 感情・コンディション記録

### US-T-020: 感情スコアを記録する 🔴 Must

**ストーリー**: 私は教員として、今日の気分を1〜5段階で記録したい。なぜなら、数値で傾向を把握しやすくしたいから。

**受け入れ基準**:
- Given: 教員がエントリ作成画面を開いている
- When: 感情スコア（1〜5）を選択して保存する
- Then: スコアがエントリに紐づいて保存され、タイムラインに表示される
- Then: 管理者にはこのスコアが集計データとして反映される（本文は非表示）

### US-T-021: 感情カテゴリを選択する 🔴 Must

**ストーリー**: 私は教員として、今の感情をカテゴリで記録したい。なぜなら、スコアだけでなく感情の種類も把握したいから。

**受け入れ基準**:
- 「喜び」「充実」「疲労」「不安」「怒り」などの感情カテゴリを選択できること
- 複数選択可能なこと

### US-T-022: 感情データを後から修正する 🟡 Should

**受け入れ基準**:
- エントリ編集時に感情スコア・カテゴリも変更できること

## EPIC-T-04: 個人ダッシュボード・可視化

### US-T-030: 感情スコアの時系列グラフを見る 🔴 Must

**ストーリー**: 私は教員として、自分の感情スコアの変化をグラフで見たい。なぜなら、数値の推移で自分のコンディションの傾向を客観的に把握したいから。

**受け入れ基準**:
- Given: 教員が個人ダッシュボードを開いている
- When: 感情グラフエリアを表示する
- Then: 過去7日間・30日間の感情スコアが折れ線グラフで表示される
- Then: グラフ表示期間を週・月で切り替えられる
- Then: 他の教員のデータは表示されない

### US-T-031: タグ別記録頻度をグラフで見る 🟡 Should

**ストーリー**: 私は教員として、どのタグの記録が多いか視覚的に確認したい。なぜなら、自分がどんな出来事に多く直面しているかを把握したいから。

**受け入れ基準**:
- タグ別の記録件数が棒グラフまたは円グラフで表示されること
- 表示期間（週・月）を切り替えられること

### US-T-032: 月次サマリーを確認する 🟡 Should

**受け入れ基準**:
- 当月の記録件数・感情スコア平均が表示されること

## EPIC-A-01: 管理者ダッシュボード

### US-A-010: 全教員のステータスを一目で確認する 🔴 Must

**ストーリー**: 私は校長として、学校全体の教員のコンディションを一画面で把握したい。なぜなら、多忙な中でも全教員の状態を素早く確認し、気になる教員を見つけたいから。

**受け入れ基準**:

- Given: 管理者がダッシュボードにログインしている
- When: 管理者ダッシュボードを表示する
- Then: テナント内の全教員がカード形式で一覧表示される
- Then: 各カードには教員名・直近の感情スコア平均・ステータスインジケーター（色分け）が表示される
- Then: 日誌の本文テキストは表示されない
- Then: 他テナント（他校）の教員は表示されない

### US-A-011: 特定教員の感情スコア推移を確認する 🔴 Must

**ストーリー**: 私は校長として、気になる教員の感情スコアの推移を詳しく見たい。なぜなら、一時的な落ち込みなのか、継続的な悪化なのかを判断したいから。

**受け入れ基準**:
- Given: 管理者が教員一覧から特定の教員カードをクリックする
- When: 教員の詳細画面が表示される
- Then: 感情スコアの時系列グラフ（週・月）が表示される
- Then: 日誌の本文テキストは表示されない

### US-A-012: 特定教員に「要注意」フラグを付ける 🟡 Should

**ストーリー**: 私は校長として、気になる教員に手動でフラグを立ててメモを残したい。なぜなら、アラートの閾値に達していなくても、直感的に気になる教員を記録しておきたいから。

**受け入れ基準**:
- 教員詳細画面から「要注意」フラグを設定できること
- フラグ設定時にメモを記入できること
- フラグが立った教員は一覧で視覚的に区別できること

## EPIC-A-02: アラートシステム

### US-A-020: アラートの発生を確認する 🔴 Must

**ストーリー**: 私は校長として、教員のコンディション悪化を示すアラートをダッシュボードで確認したい。なぜなら、自動検知によって見落としのないサポートができるようにしたいから。

**受け入れ基準**:

- Given: 教員の感情スコアが設定した閾値を連続して下回っている
- When: システムがアラートを自動生成する
- Then: 管理者ダッシュボードのアラート件数バッジが更新される
- Then: アラート一覧に対象教員名・発生日時・アラート種別が表示される
- Then: 日誌の本文テキストはアラート情報に含まれない

- Given: 教員が設定日数以上記録を途絶えさせている
- When: システムがアラートを自動生成する
- Then: 同様にアラート一覧に表示される

### US-A-021: アラートを「対応済み」にクローズする 🔴 Must

**ストーリー**: 私は校長として、対応済みのアラートをクローズしたい。なぜなら、未対応のアラートと対応済みのアラートを区別して管理したいから。

**受け入れ基準**:
- Given: 管理者がアラートに対応した
- When: アラートを「対応済み」にする
- Then: アクティブなアラート一覧から消え、履歴に移動する
- Then: クローズした管理者名と日時が記録される

### US-A-022: アラート履歴を確認する 🟡 Should

**ストーリー**: 私は校長として、過去のアラート対応履歴を参照したい。なぜなら、教員のサポート経緯を振り返りたいから。

**受け入れ基準**:
- 対応済みのアラートが一覧で参照できること
- 発生日時・対応日時・対応者が記録されていること

## EPIC-S-01: テナント管理

### US-S-001: 新規学校テナントを作成する 🔴 Must

**ストーリー**: 私はシステム管理者として、新しく契約した学校のテナントを作成したい。なぜなら、その学校の教員・管理者がすぐにサービスを利用できるようにしたいから。

- テナント停止後、そのテナントのユーザーはログインできなくなること
- テナントのデータは一定期間保持されること（即時削除しない）

## ストーリー変更履歴

### 2026-04-15: Unit-02 機能設計フェーズでの設計詳細化を反映

**背景**: Unit-02 の機能設計フェーズで業務分析を深めた結果、当初の「個人タイムライン」モデルから「テナント内共有タイムライン + 個人記録」の2面構成に進化した。同一学校内の教員が互いに学び合える価値を優先した判断。

**影響したストーリー**:

1. **US-T-014（タイムラインで日誌を閲覧する）**
   - 旧: 「自分のエントリのみが新しい順に表示される、他の教員のエントリは表示されない」
   - 新: 「同一テナントの全教員の **公開エントリ** が新しい順に表示される（共有タイムライン）、自分の全エントリは別途マイ記録（`/journal/mine`）で閲覧」
   - 理由: 教員間で気づきや工夫を共有して互いに学び合う価値を提供するため
   - 関連設計: `aidlc-docs/construction/unit-02/functional-design/business-logic-model.md` BP-J-04a/04b

2. **US-T-010（日誌エントリを作成する）**
   - 旧: 「タイムラインの先頭に新しいエントリが表示される」
   - 新: 「マイ記録の先頭に表示される。`is_public=true` のチェックを外していなければ共有タイムラインにも表示される。チェックを入れた場合（自分だけに保存）はマイ記録のみに残る」
   - 理由: 共有タイムラインの導入に伴い、デフォルト公開・任意で非公開の選択肢を追加
   - 200文字制限の明文化を追加

3. **US-T-013（エントリにタグを付ける）**
   - 旧: 「複数のタグを付与」「既存タグから選択 or 新規作成」
   - 新: 「最大10件まで」「感情タグ・業務タグの統合（is_emotion フラグ）」「20件超でフィルタ」「テナント作成時にデフォルト8件シード」
   - 理由: タグ統合（旧 `EmotionCategory` を `tags.is_emotion` に統合）と NFR-U02-01/03 の仕様確定を反映

4. **US-T-011・US-T-012**
   - 受け入れ基準を実装の細部に合わせて拡充（URL 直打ち防御・CASCADE 削除の挙動など）

**未影響のストーリー**:
- US-T-020/021/022（感情記録）: タグ統合に伴い表現は変わるが、ユーザー体験としての受け入れ基準は維持
- US-S-001/002 等の管理者向けストーリー: Unit-02 では未変更

**E2E テストへの紐づけ**: 本変更後の受け入れ基準が `__tests__/e2e/*.spec.ts`（Step 16b）の Oracle として使用される。

**設計ドリフト発見の経緯**: Step 16b（E2E）計画時、ユーザーから「E2E は受け入れ基準をテストする認識でよいか」という確認を受け、stories.md と現実装の比較を実施した結果、上記の乖離を発見・反映した。
