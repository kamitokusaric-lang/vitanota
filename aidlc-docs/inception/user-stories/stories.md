# vitanota — ユーザーストーリー

## 整理方針
- **構造**: ペルソナ × エピック（大機能単位）× ストーリー
- **優先度**: MoSCoW法（Must / Should / Could / Won't）
- **受け入れ基準**: 主要ストーリーは Given/When/Then、シンプルなものは箇条書き
- **言語**: 日本語

## MoSCoW 凡例
| 記号 | 意味 |
|---|---|
| 🔴 Must | MVPに必須。これがなければリリースできない |
| 🟡 Should | 重要だが、MVP直後のリリースでも許容できる |
| 🟢 Could | あれば価値があるが、なくてもよい |
| ⚫ Won't | フェーズ2以降に実装する |

## EPIC-T-01: 認証・アカウント

### US-T-001: Googleアカウントでログインする 🔴 Must

**ストーリー**: 私は教員として、Googleアカウントでログインしたい。なぜなら、別途パスワードを管理する手間なく安全にサービスを利用したいから。

**受け入れ基準**:
- GoogleログインボタンをクリックするとGoogle認証画面に遷移すること
- 認証成功後、自分のテナント（学校）の教員ダッシュボードにリダイレクトされること
- 他校のデータにはアクセスできないこと
- 認証失敗時は適切なエラーメッセージが表示されること

### US-T-002: ログアウトする 🔴 Must

**ストーリー**: 私は教員として、セキュアにログアウトしたい。なぜなら、共有端末での情報漏洩を防ぎたいから。

**受け入れ基準**:
- ログアウト後、セッションが即座に無効化されること
- ログアウト後にブラウザの戻るボタンを押しても認証済みページにアクセスできないこと

### US-T-003: AI週次レポートへの同意を設定する 🟡 Should

**ストーリー**: 私は教員として、AIレポート機能への同意・拒否を設定したい。なぜなら、自分のデータがAI処理に使われることを理解したうえで判断したいから。

**受け入れ基準**:
- 初回ログイン時に同意確認画面が表示されること
- 同意しない場合でも、AI週次レポート以外の機能は全て利用できること
- 設定はいつでも変更できること

## EPIC-T-02: 日誌記録

### US-T-010: 日誌エントリを作成する 🔴 Must

**ストーリー**: 私は教員として、今日の出来事や気持ちをプレーンテキストで記録したい。なぜなら、日々の変化を後から振り返れるようにしたいから。

**受け入れ基準**（2026-04-15 改訂: 共有タイムライン方式・巻末の変更履歴を参照）:
- Given: 教員が `/journal` にログインしている
- When: 「新規投稿」ボタンをクリックしてテキストを入力し保存する
- Then: マイ記録（`/journal/mine`）の先頭に新しいエントリが表示され、作成日時が記録される
- Then: 「自分だけに保存」を**チェックしなかった**場合、共有タイムライン（`/journal`）の先頭にも表示され、テナント内の他教員からも閲覧可能となる
- Then: 「自分だけに保存」を**チェックした**場合、共有タイムラインには表示されずマイ記録のみに残る

- Given: 1日に複数回記録したい
- When: 同じ日に2件目のエントリを作成する
- Then: 同じ日付で複数のエントリが時系列で表示される

- Given: 本文は 1〜200 文字の範囲で入力する
- When: 空文字もしくは 201 文字以上で保存しようとする
- Then: クライアント側バリデーションでエラー表示され、送信されない

### US-T-011: 日誌エントリを編集する 🔴 Must

**ストーリー**: 私は教員として、記録した内容を後から修正したい。なぜなら、書き間違いや書き足しが発生するから。

**受け入れ基準**:
- 自分のエントリのみ編集できること（他の教員のエントリは編集不可、URL 直打ちでも 404 が返る）
- 編集後、`updated_at` が現在時刻に更新されること
- 編集ページ（`/journal/[id]/edit`）はマイ記録からの編集リンク経由でのみ到達する導線
- 公開設定（is_public）も編集時に変更可能

### US-T-012: 日誌エントリを削除する 🔴 Must

**ストーリー**: 私は教員として、記録したエントリを削除したい。なぜなら、誤って記録した内容を残したくないから。

**受け入れ基準**:
- 削除前にブラウザの `confirm()` ダイアログで確認を求めること
- 自分のエントリのみ削除できること（他の教員のエントリは URL 直打ちでも 404）
- 削除後、エントリに紐づいていた `journal_entry_tags` も自動削除されること（タグ本体は残る）

### US-T-013: エントリにタグを付ける 🔴 Must

**ストーリー**: 私は教員として、エントリにタグを付けて分類したい。なぜなら、後からどんな出来事が多かったかを把握したいから。

**受け入れ基準**（2026-04-15 改訂: タグ統合・巻末の変更履歴を参照）:
- 1つのエントリに最大10件までのタグを付与できること
- 既存タグから選択できること（テナント内の全タグがボタン表示される）
- タグには感情タグ（`is_emotion=true`）と業務タグ（`is_emotion=false`）があり、UI 上で視覚的に区別されること
- テナント作成時にシステムデフォルトタグ8件（感情5・業務3）が自動シードされていること
- タグが20件を超えた場合、テキスト入力による絞り込みができること
- 新しいタグを教員が作成できること（school_admin のみが削除可能）

### US-T-014: タイムラインで日誌を閲覧する 🔴 Must

**ストーリー**: 私は教員として、テナント内の教員全員で日誌をタイムライン形式で共有閲覧したい。なぜなら、同じ学校内での気づきや工夫を共有して、互いに学び合いたいから。

**受け入れ基準**（2026-04-15 改訂: 共有タイムライン方式に更新・巻末の変更履歴を参照）:

**共有タイムライン（`/journal`）**:
- Given: 教員が `/journal` を開いている
- When: タイムラインが表示される
- Then: 同一テナントの全教員の **公開エントリ**（`is_public=true`）が新しい順に20件/ページで表示される
- Then: 各エントリは投稿者名・日時・本文冒頭50文字・タグを表示する
- Then: **非公開エントリ（`is_public=false`）は絶対に表示されない**（SP-U02-04 8層防御）
- Then: 別テナントの教員のエントリは一切表示されない（RLS + 複合 FK）

**マイ記録（`/journal/mine`）**:
- Given: 教員が `/journal/mine` を開いている
- When: マイ記録が表示される
- Then: 自分の全エントリ（公開・非公開両方）が新しい順に表示される
- Then: 非公開エントリには「自分だけに保存」バッジが表示される
- Then: 各エントリから編集ページに遷移できる

## EPIC-A-01: 管理者ダッシュボード

### US-A-010: 全教員のステータスを一目で確認する 🔴 Must

**ストーリー**: 私は校長として、学校全体の教員のコンディションを一画面で把握したい。なぜなら、多忙な中でも全教員の状態を素早く確認し、気になる教員を見つけたいから。

**受け入れ基準**:

- Given: 管理者がダッシュボードにログインしている
- When: 管理者ダッシュボードを表示する
- Then: テナント内の全教員がカード形式で一覧表示される
- Then: 各カードには教員名・直近の感情スコア平均・ステータスインジケーター（色分け）が表示される
- Then: 日誌の本文テキストは表示されない
- Then: 他テナント（他校）の教員は表示されない

### US-A-011: 特定教員の感情スコア推移を確認する 🔴 Must

**ストーリー**: 私は校長として、気になる教員の感情スコアの推移を詳しく見たい。なぜなら、一時的な落ち込みなのか、継続的な悪化なのかを判断したいから。

**受け入れ基準**:
- Given: 管理者が教員一覧から特定の教員カードをクリックする
- When: 教員の詳細画面が表示される
- Then: 感情スコアの時系列グラフ（週・月）が表示される
- Then: 日誌の本文テキストは表示されない

### US-A-012: 特定教員に「要注意」フラグを付ける 🟡 Should

**ストーリー**: 私は校長として、気になる教員に手動でフラグを立ててメモを残したい。なぜなら、アラートの閾値に達していなくても、直感的に気になる教員を記録しておきたいから。

**受け入れ基準**:
- 教員詳細画面から「要注意」フラグを設定できること
- フラグ設定時にメモを記入できること
- フラグが立った教員は一覧で視覚的に区別できること

## EPIC-S-01: テナント管理

### US-S-001: 新規学校テナントを作成する 🔴 Must

**ストーリー**: 私はシステム管理者として、新しく契約した学校のテナントを作成したい。なぜなら、その学校の教員・管理者がすぐにサービスを利用できるようにしたいから。

**受け入れ基準**:
- テナント作成時に学校名・管理者メールアドレスを設定できること
- テナント作成後、他の既存テナントのデータに影響がないこと（隔離確認）
- 新テナントの管理者が初回ログインできること

### US-T-098: マイ記録をエクスポートする 🔴 Must

**ストーリー**: 私は教員として、自分のマイ記録を JSON / Markdown 形式でダウンロードしたい。なぜなら、転勤・退会の際に自分の振り返りを手元に残したいから、また個人情報保護法のデータポータビリティ権に対応するため。

**受け入れ基準**:
- Given: 在籍中の教員が `/me/export` 画面を開いている
- When: 「マイ記録をダウンロード」ボタンをクリックする
- Then: 自分の全エントリ（公開・非公開両方）が JSON ファイルとしてダウンロードされる
- Then: JSON にはエントリ ID・本文・公開フラグ・作成日時・更新日時・タグ名一覧を含む
- Then: 他人のエントリ・他テナントのエントリは絶対に含まれない（RLS で保証）

- Given: 同じ画面で「Markdown 形式」を選択する
- When: ダウンロードボタンをクリックする
- Then: 人間可読な Markdown ファイルがダウンロードされる
- Then: 各エントリは見出し・本文・タグ・日付の構造で出力される

- Given: 在籍中はいつでもエクスポート可能
- When: 月次・年次など自由なタイミングで実行する
- Then: 制限なくダウンロードできる（rate limit はあり）

- Given: エクスポート操作は監査ログに記録される
- Then: `LogEvents.UserExported` イベントが発火される

**実装フェーズ**: 次 Unit（API・Service の実装）・Phase 1 はスキーマ準備のみ

### US-T-099: vitanota を退会する 🔴 Must

**ストーリー**: 私は教員として、vitanota の利用を終了したい。なぜなら、別サービスへの移行や教職を離れる場合に、自分の個人情報を削除する必要があるから。

**受け入れ基準**:
- Given: 教員が `/me/withdraw` 画面を開いている
- When: 退会フォームに記入する
- Then: 退会理由を任意で記入できる
- Then: 退会前に「マイ記録をダウンロード」を実行できる動線がある（US-T-098 へ）
- Then: 「公開エントリの扱い」を選択できる:
  - A. 匿名化して残す（推奨）
  - B. 全削除（個人情報削除請求）

- When: 退会を確定する
- Then: `users.deleted_at = NOW()` が設定される（soft delete）
- Then: 全 `sessions` が即時 DELETE される（即時ログアウト）
- Then: 全 `accounts` 行が DELETE される（再ログイン不可）
- Then: 全 `user_tenant_roles` が DELETE される
- Then: マイ記録（`is_public=false`）は grace period 中（30日）保持
- Then: 公開エントリ（選択 A）は `user_id = NULL` に更新（匿名化）
- Then: 公開エントリ（選択 B）は DELETE
- Then: ブラウザは即座にログイン画面へリダイレクト

- Given: 退会後 30日以内
- When: school_admin に復旧を依頼する
- Then: `users.deleted_at = NULL` に戻すことで復旧可能（手動 SQL Runbook）

- Given: 30日経過後
- When: 物理削除バッチが走る
- Then: `DELETE FROM users WHERE deleted_at < NOW() - INTERVAL '30 days'`
- Then: CASCADE で残った関連データも完全削除

**実装フェーズ**: 次 Unit

### US-T-100: 学校から離脱する（転勤） 🟡 Should

**ストーリー**: 私は教員として、転勤や退職に伴い1つの学校テナントから外れたい。ただし vitanota 自体は別の学校で使い続ける可能性がある。

**受け入れ基準**:
- Given: 教員 X が学校 A の `user_tenant_roles` に所属している
- When: school_admin (A) が「教員を学校から外す」操作を実行する
- Then: `DELETE FROM user_tenant_roles WHERE user_id=X AND tenant_id=A`
- Then: 教員 X の学校 A での認証コンテキストは消失（次のリクエストで 403）
- Then: 教員 X が他テナントに所属していれば、そのテナントでは引き続き利用可能
- Then: 学校 A の共有タイムラインに残っていた教員 X の公開エントリは `user_id = NULL` に更新（匿名化、Q1-B）
- Then: 学校 A の教員 X のマイ記録は grace period 中保持・30日後に削除
- Then: 教員 X の `users` 行は削除されない（兼務復帰の可能性を残す）

- Given: 学校 A で教員 X のマイ記録をエクスポートしてから転勤する場合
- When: 退職前に教員 X が US-T-098 を実行
- Then: マイ記録がダウンロードされる
- Then: 教員 X 個人がデータを保持

**実装フェーズ**: 次 Unit（school_admin の管理画面と統合）

### US-S-002: テナントを停止する 🟡 Should

**ストーリー**: 私はシステム管理者として、解約した学校のテナントを停止したい。なぜなら、契約終了後のアクセスを遮断したいから。

**受け入れ基準**:
- テナント停止後、そのテナントのユーザーはログインできなくなること
- テナントのデータは一定期間保持されること（即時削除しない）

### US-S-003: school_admin が教員を学校から外す 🔴 Must

**ストーリー**: 私は school_admin として、退職・転勤した教員を自分の学校から外したい。なぜなら、退職後のアクセスを遮断し、新しい所属者リストを正しく管理するため。

**受け入れ基準**:
- Given: school_admin が教員一覧画面を開いている
- When: 該当教員の「学校から外す」を実行する
- Then: 確認ダイアログが表示される
- Then: 確定すると `DELETE FROM user_tenant_roles WHERE user_id=? AND tenant_id=?`
- Then: 該当教員の学校 A セッションは即時無効化（`DELETE FROM sessions WHERE user_id=? AND active_tenant_id=?`）
- Then: 公開エントリは匿名化（`UPDATE journal_entries SET user_id=NULL WHERE user_id=? AND tenant_id=?`）
- Then: マイ記録は grace period 中保持
- Then: 監査ログに `LogEvents.UserTransferredFromTenant` を記録（誰が誰をいつ外したか）
- Then: 教員本人の `users` 行は触らない（他テナント所属の可能性）

**実装フェーズ**: 次 Unit（admin 機能）

### US-S-004: 退会済みユーザーを物理削除する（30日後バッチ） 🟢 Could

**ストーリー**: 私は system 管理者として、退会後 30日経過したユーザーを物理削除したい。なぜなら、個人情報保護法の保有期間最小化の原則に従うため。

**受け入れ基準**:
- Given: 日次バッチが起動
- When: `DELETE FROM users WHERE deleted_at IS NOT NULL AND deleted_at < NOW() - INTERVAL '30 days'`
- Then: CASCADE で残った関連データも全削除（マイ記録・タグ created_by 等は SET NULL）
- Then: 監査ログに `LogEvents.UserHardDeleted` を記録
- Then: 削除前のメタデータ（userId・退会日時）はログにのみ残る

**実装フェーズ**: 次 Unit（Lambda マイグレーター基盤を流用）
